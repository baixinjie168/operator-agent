1. 背景：文档特殊性与当前症状

1.1 文档结构

operators/trans-ten/aclnnNpuFormatCast.md 比常规算子文档多了 一个独立的预处理函数 aclnnNpuFormatCastCalculateSizeAndFormat，共 3 个函数：

函数参数说明
aclnnNpuFormatCastCalculateSizeAndFormatsrcTensor, dstFormat, additionalDtype, dstShape, dstShapeSize, actualFormat预处理：计算目标张量的 shape 和实际数据格式
aclnnNpuFormatCastGetWorkspaceSizesrcTensor, dstTensor, workspaceSize, executor两段式接口第一段（常规）
aclnnNpuFormatCastworkspace, workspaceSize, executor, stream两段式接口第二段（常规）

每个函数在文档中都有独立的 ## 函数名 H2 章节和 - **参数说明** H3 子章节（含 HTML 参数表）。

1.2 当前症状

运行 extract 流水线后，res/check0627/cons/aclnnNpuFormatCast.json 的结果：

当前输出（缺失） 
inputs:  srcTensor, quantization_type
outputs: dstTensor
缺失的 CalculateSizeAndFormat 参数：
  dstFormat, additionalDtype (inputs)
  dstShape, dstShapeSize, actualFormat (outputs)
期望输出（同 FFNV3 结构） 
inputs:  srcTensor, dstFormat, additionalDtype,
          quantization_type (+ 隐式参数 C0)
outputs: dstTensor, dstShape, dstShapeSize, actualFormat
顶层 key 结构与 aclnnFFNV3.json 一致

值得注意的是：dtype_support_description 字段中已经包含 aclnnNpuFormatCastCalculateSizeAndFormat.* 的条目——说明 dtype 组合被正确提取了，只是参数没有被放入 inputs/outputs。

2. 根因分析

通过追踪完整解析流水线代码，定位到 4 个根因，它们共同导致 CalculateSizeAndFormat 函数的参数全部丢失。

根因 1：章节分类错误（section_splitter.py）

_classify_h2() 只识别三类 H2 标题：

if heading.endswith("GetWorkspaceSize"):  # GET_WORKSPACE_SIZE
if heading == operator_name:              # EXECUTE_API
if 匹配 H2_PATTERNS(...):                 # 产品支持/功能说明/...
else:                                     # UNKNOWN  <- CalculateSizeAndFormat 落入此处

aclnnNpuFormatCastCalculateSizeAndFormat 不以 GetWorkspaceSize 结尾、不等于算子名、不匹配任何 H2 模式 -> 分类为 UNKNOWN。

更严重的是，它的 H3 子章节 参数说明 被 _classify_h3() 分类为 params_get_workspace（因为父类型不是 EXECUTE_API）。

用实际文档运行 split_sections() 确认：

# 实际章节划分结果（13 个 section）
[3] params_get_workspace        # CalculateSizeAndFormat 的参数表（错误分类！）
[4] return_codes_get_workspace  # CalculateSizeAndFormat 的返回值表
[5] get_workspace_size          # GetWorkspaceSize H2
[6] params_get_workspace        # GetWorkspaceSize 的参数表（被同名遮蔽！）
[7] return_codes_get_workspace  # GetWorkspaceSize 的返回值表
[8] execute_api / [9] params_execute / [10] return_codes_execute ...

根因 2：get_section_by_type 只返回第一个匹配（document_tools.py）

def get_section_by_type(doc_id, section_type):
    for section in parsed["sections"]:
        if section["section_type"] == section_type:
            return section  # 只返回第一个匹配！
    return None

根因 1 产生了两个 params_get_workspace 章节，调用 get_section(doc_id, "params_get_workspace") 只返回 CalculateSizeAndFormat 的参数表（排在文档前面），GetWorkspaceSize 的参数表被完全遮蔽，下游永远看不到。

根因 3：_build_inputs_outputs 只保留 WorkspaceSize 函数参数（assemble_result.py）

# assemble_result.py _build_inputs_outputs()
for p in params:
    fn = p.get("function_name", "")
    if not fn.endswith("WorkspaceSize"):  # CalculateSizeAndFormat 被排除！
        continue

即便 CalculateSizeAndFormat 的参数被提取到 DB，最终组装 inputs/outputs 时也被过滤掉。当前输出验证：

  • INPUTS: 只有 srcTensor + quantization_type（缺失 dstFormat, additionalDtype）
  • OUTPUTS: 只有 dstTensor（缺失 dstShape, dstShapeSize, actualFormat）

根因 4：多个节点按 ws/exe 二分分组，CalculateSizeAndFormat 参数被归入 exe 组

以下节点用 is_ws_function()（判断函数名是否含 GetWorkspaceSize）做二分组，CalculateSizeAndFormat 参数被归入 exe 组，用 params_execute 章节提取（错误章节）：

3. 解决方案总览

核心思路

为 CalculateSizeAndFormat 类函数建立 独立的章节类型和函数分组，让它贯穿整个解析流水线。不需要新增节点或调整节点顺序——所有改动都在现有节点内部完成。

数据流变化：
现状: 文档 -> UNKNOWN H2 -> params_get_workspace (遮蔽) -> 只剩 srcTensor
目标: 文档 -> CALCULATE_SIZE_API H2 -> params_calculate_size (独立) -> 完整参数

三层修复链路：

  1. 章节类型分离（阶段 1-2）：CalculateSizeAndFormat 获得独立 section_type，不再遮蔽 GetWorkspaceSize
  2. 函数感知匹配（阶段 3-4）：各节点 fetch 正确章节，table_column_extract 按函数名匹配参数表
  3. 组装过滤放宽（阶段 5）：_build_inputs_outputs 接纳 CalculateSizeAndFormat 参数

7. 阶段 4：table_column_extract 函数感知匹配（核心改动）

问题

当前 _extract_from_sections 把所有章节文本拼接后按参数名匹配（取第一个匹配）。srcTensor 同时出现在 CalculateSizeAndFormat 和 GetWorkspaceSize 两张表中，会导致跨函数错配。

方案：按 section_type 分组匹配

每个参数只在自己所属函数的表中查找。重构 _extract_from_sections：

新增：函数名 -> 参数表 section_type 映射

def _function_to_param_section(fn: str) -> str:
    if "CalculateSize" in fn:
        return "params_calculate_size"
    if "GetWorkspaceSize" in fn:
        return "params_get_workspace"
    return "params_execute"

重构 _fetch_sections_text -> _fetch_sections（返回列表）

async def _fetch_sections(doc_id) -> list[tuple[str, str]]:
    sections = []
    for st in ["params_calculate_size", "params_get_workspace", "params_execute"]:
        section = await _mcp_client.get_section(doc_id, st)
        if section and section.get("content"):
            sections.append((st, section["content"]))
    return sections

11. 多场景处理说明

文档的"约束说明"章节包含 <details> 块，按平台分别列出不同的 dtype/format 组合和 C0 计算方法：

平台dstFormat 支持C0 计算方法
Atlas 350 加速卡仅 ACL_FORMAT_FRACTAL_NZ(29)C0 = 32B / size of additionalDtype
Atlas A2 / A3ND, NZ, NCDHW, NDC1HWC0, FRACTAL_Z_3DC0 = 32B / size of srcTensor 基础类型

这部分由 dtype_combo_extract（LLM）和 constraint_extract（5 Pass）处理，已经能正确提取（当前输出可见 dtype_support_description 有正确的 per-platform 条目）。改动后 CalculateSizeAndFormat 参数进入 inputs/outputs，这些约束就能引用到已定义的参数了。

另外，C0 作为隐式参数（计算值）目前在 constraints 中被引用但未定义在 inputs 中，这是次要问题，可在本方案落地后单独处理。

    if calc_content.strip():
        if ws_content.strip():
            ws_content = calc_content + "\n\n" + ws_content
        else:
            ws_content = calc_content  # 仅 calc 有内容时也作为 ws 主内容
    exe_content = exe_section.get("content", "") if exe_section else ""
    constraints_text = (
        constraints_section.get("content", "") if constraints_section else ""
    )

    # 单函数提升（原有逻辑不变）
    if not ws_content.strip() and exe_content.strip():
        ws_content = exe_content
        exe_content = ""

    if constraints_text:
        ws_content += _CONSTRAINTS_PREFIX + constraints_text

    return ws_content, exe_content, constraints_text
