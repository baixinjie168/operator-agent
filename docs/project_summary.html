项目定位

基于 LangGraph + MCP 的 CANN 算子文档自动化处理系统。将华为昇腾平台的算子 Markdown 文档解析为结构化数据，经 LLM 提取参数、约束、产品支持信息，最终生成 Python 测试用例。

当前完成度：约 93%，文档解析、数据库层、Pipeline 全部 20 个节点（含 table_column_extract 表格直提节点、llm_description_extract 子图（预过滤 + 属性校验 + verify_and_enhance 审核增强 + is_support_discontinuous 保护）、function_signature 提取（含参数列表生成 + 类型后处理规范化）、function_explanation 提取、shape/dtype/dformat/optional/array_length 提取（含表格直提跳过逻辑）、allowed_range 提取、return_code 提取、determinism 提取、dtype_combo 提取、参数关系提取子图（implicit_param_extract 隐式参数提取 + platform_constants 平台外部常量 + parameter_representation_build 确定性表示关系生成 + Agent 迭代循环：分块提取 + 双重覆盖率 + 针对性提取 + 自反思 + 隐式参数 prompt 增强）、build_param_relations 参数关系表达式构建节点（三层防护：AST校验 + Few-shot + 语义验证 + Phase 5 常量值替换 + 隐式参数白名单）、build_single_param_constraint 单参数约束提取节点（两层防护：确定性正则匹配 ~93% 覆盖 + LLM 长尾提取 ~7%，追加到 param_relations 表，配套 single_param_prompt.py 提供 Prompt 和解析器）、build_param_constraint 参数约束构建子图（fetch_param_data → [dimensions_build ∥ allowed_range_build ∥ attrs_build] → constraint_assemble，含类型规范化 + dimensions rank 解析修复 + is_operator_param 隐式参数标记）、assemble_result 汇编节点 + json_constraints 写入）可运行。数据质量问题（dtype key 不匹配、const 限定符污染、符号维度解析错误）已修复。人工审核、测试用例生成尚未实现。

fetch_sections

从 parameters 中按 function_name 分为 ws/exe 两组，仅对有参数的组调用 MCP 获取 section 内容，避免不必要的 MCP 调用。

extract_ws / extract_exe

  • 预过滤上下文：extract_param_context(sections_text, param_name) 将完整 section 文本（3000-12500 字符）缩减为仅与该参数相关的片段（500-2000 字符），减少 LLM 注意力分散

  • 一次 LLM 调用提取四个字段：llm_description + src_content + direction + is_support_discontinuous

  • direction 保护：如果上游 table_column_extract 已直提 direction（表格形态参数），_extract_one() 保留原值不覆盖

validate_results

对提取结果执行属性检查清单校验（attribute_checklist_verify），检查 llm_description 是否覆盖了上下文中实际存在的属性（direction/dtype/shape/optional/discontinuous）。同时生成覆盖率报告（coverage_report），统计提取成功数、缺失数、描述过短数等指标。

save_descriptions

llm_description 完整性保障（verify_and_enhance）

方案来源：docs/llm_description-completeness.html | 已实施

核心问题

llm_description 是 6 个下游节点（shape/dtype/dformat/optional/array_length/build_param_constraint）的唯一输入。一旦描述丢失信息，6 个节点级联失败。本方案在子图内插入 verify_and_enhance 节点，在源头修复描述质量问题。

设计原则

  • 源头修复：llm_description 在子图内就变得完整，下游零改动
  • 精确触发：仅对有 missing_attrs 或 desc_too_short 的参数调用 LLM（~15% 参数）
  • 用文档事实说话：LLM 审核员对照原始文档判断，文档没有的信息不会强加
  • 全程可追溯：description_audit JSON 字段记录提取→校验→审核的完整过程

verify_and_enhance 节点逻辑

verify_and_enhance_node(state) │ ├─ 筛选需要审核的参数（满足任一即触发）: │ ├─ _validation.missing_attrs 非空 │ └─ _validation.desc_too_short == True │ ├─ 逐参数并发审核（Semaphore(5)）: │ ├─ 输入: 原始描述 + 缺失属性列表 + 文档上下文 │ ├─ LLM: LLM_DESCRIPTION_VERIFY_PROMPT │ │ └─ 输出: {has_missing_info, found_attrs, enhanced_description} │ │ │ ├─ 决策: │ │ ├─ has_missing_info=true 且 enhanced 更长 → 替换 llm_description │ │ └─ 否则 → 保留原结果 │ │ │ └─ 构建 description_audit 记录: │ {extraction: {original_description, char_count}, │ validation: {triggered, missing_attrs}, │ verification: {triggered, has_missing_info, enhanced}} │ └─ 返回更新后的 ws_results / exe_results + enhance_count

description_audit 字段

参数关系提取 Agent 循环详解

方案来源：docs/param_relation-schema.html | 已实施

核心问题

原有实现为 ws 和 exe 各 1 次 LLM 调用，一次性提取所有参数耦合关系。这种方式存在三大风险：

风险说明
一次性提取遗漏LLM 倾向提取"明显的"表格关系，忽略散文中的隐性关系；输出超过 5-8 项后质量下降；跨 section 信息距离远时关系丢失
归因错误长表格中 LLM 可能将第 N 行关系错误归因到相邻行
跨 section 丢失params section + constraints section 拼接后，相关信息距离远，LLM 无法关联

Agent 循环通过分治策略突破一次性提取的召回上限，预期召回率从 ~65% 提升至 ~92%。

方案来源：docs/expr-schema.html | 已实施

核心问题

build_param_relations 是项目中唯一将自然语言转化为可执行 Python 表达式的节点。当前实现为单次 LLM 调用 + 零校验，存在幻觉参数、幻觉属性、条件方向反转、量词混淆等风险。表达式错误会静默存入数据库并出现在最终 JSON Constraint 输出中。

三层防护架构

所有优化逻辑在 build_param_relations_node 内部实现，不新增 Node 或子图，不改 graph.py。

确定性预处理 + 校验机制（2026-06-08 新增）：对 dimensions 和 allowed_range_value 实施两阶段保护：

  1. Phase 1 - 确定性预处理：基于正则的零 LLM 成本解析

      • 逐维范围：[[min,max], ...] - 每维的 size 范围（如 "[2, 3, 4]" → [[2,2],[3,3],[4,4]]）
    • allowed_range_value：常见模式直接提取
      • 范围："0-100" / "[1, 8]" → [[0, 100]] / [[1, 8]]
      • 枚举："枚举值: 1, 2, 3" → [[1,1], [2,2], [3,3]]
      • bool 类型短路：[True, False]

  2. Phase 2 - LLM 解析 + 校验 + 重试
    • 确定性预处理未匹配的 shape 进入 LLM 批量解析（SHAPE_TO_DIMENSIONS_PROMPT）
    • 确定性预处理未匹配的 allowed_range_value 进入 LLM 逐参数提取（ALLOWED_RANGE_VALUE_BUILD_PROMPT）

    • 对齐校验（dimensions 专用）：LLM 返回数组长度必须与输入 shape 数量一致

MCP 通信模式：每个节点每次调用 MCP 工具时，MCPClient._call_tool() 会启动一个新的 MCP Server 子进程（async with stdio_client），调用完成后进程退出。整个 Pipeline 执行一次约产生 ~60 次子进程启动（llm_description_extract 子图额外增加 5~7 次 [fetch + N 个参数的 update]，table_column_extract 额外增加 4~7 次 [3 个 get_section + 4 个 batch update]，param_relation_extract 子图额外增加 6~8 次 [3 个 get_section + query_function_signatures + query_platform_support + save_implicit_params + save_platform_constants + save_parameter_representations + save_param_relations]，allowed_range_extract 额外增加 3~6 次，return_code_extract 额外增加 3~6 次，determinism_extract 额外增加 2~3 次，dtype_combo_extract 额外增加 2~3 次，build_param_relations 额外增加 3~5 次，build_single_param_constraint 额外增加 2~4 次 [query_params + query_param_relations + save_param_relations]，build_param_constraint 子图额外增加 5~8 次 [fetch_param_data 查询 + dimensions/range/attrs 各自的查询]，assemble_result 额外增加 10 次）。参数列表在 function_signature_extract 中创建，table_column_extract 直提表格值后，llm_description_extract 充实描述字段并通过 state 传递给下游，避免重复 MCP 查询。隐式参数提取（BS/H/N/rankSize 等）在 param_relation_extract 子图内的 implicit_param_extract 节点完成。
