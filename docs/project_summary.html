Operator Agent 项目总览

生成日期: 2026-05-28 | 更新日期: 2026-05-30 | 基于代码实际状态

1. 项目定位

基于 LangGraph + MCP 的 CANN 算子文档自动化处理系统。将华为昇腾平台的算子 Markdown 文档解析为结构化数据，经 LLM 提取参数、约束、产品支持信息，最终生成 Python 测试用例。

当前完成度：约 50%，文档解析、数据库层、Pipeline 全部 8 个节点（含 src_content 提取、shape/dtype/optional 提取）可运行。约束提取、人工审核、测试用例生成尚未实现。

5.2 各节点详细职责

节点1: init_doc_node（顺序执行）

触发条件：Pipeline 启动后第一个执行。

init_doc_node(state) │ ├─ MCP: check_version(operator_name, content_hash) │ └─ 查询数据库，返回 {status: "new"|"unchanged"|"updated", version, doc_id} │ ├─ 如果 status == "unchanged"（内容完全未变）: │ ├─ MCP: get_parsed(operator_name, existing_version) │ └─ 返回 {status="unchanged", doc_id, sections, cann_version, error=None} │ └─ 如果 status == "new" 或 "updated": ├─ MCP: save_doc(operator_name, content) → {operator_id, version, doc_id} ├─ MCP: parse_doc(content) → ParsedOperatorDocument ├─ MCP: save_parsed(operator_name, version, parsed_dict) └─ 返回 {status, version, doc_id, cann_version, sections, error=None}

输出状态：operator_id, doc_id, version, sections, cann_version, status, error

路由函数：_should_continue

def _should_continue(state: dict) -> list[str]:
    if state.get("status") in ("error",):
        return END                    # 终止 Pipeline
    return ["product_support", "parse_params"]  # 扇出到两个并行节点
⚠ status=="unchanged" 时仍扇出到并行节点，导致不必要的 MCP 重复调用。应在此处加入 "unchanged" 判断直接终止 Pipeline。

节点2a: product_support_node（与 2b 并行执行）

流程：MCP 获取文档 → 查找 product_support section → LLM 提取产品支持表 → MCP 保存到 document_versions.product_support。

输出状态：product_support: list[dict], error

节点2b: parse_params_node（与 2a 并行执行）

流程：MCP 获取文档 → 查找 function_prototype section → LLM 提取函数参数列表 → 展开为参数记录 → MCP 保存到 parameters 表。

输出状态：parameters: list[dict], error

节点3: src_content_extract_node（顺序执行，等待 2a 和 2b 都完成）

流程：

src_content_extract_node(state) │ ├─ MCP: query_params_by_doc_id(doc_id) → 当前版本的参数列表 ├─ MCP: get_section(doc_id, "params_get_workspace") + get_section("params_execute") │ ├─ 按函数名拆分：GetWorkspaceSize → ws_content, Execute → exe_content │ ├─ 并发 LLM（asyncio.Semaphore(5)）: │ └─ SRC_CONTENT_EXTRACT_PROMPT.format(param_name, content) │ └─ 从文档章节中提取参数的原始描述文本 │ ├─ MCP: update_param_src_content(doc_id, updates) │ └─ 批量 UPDATE parameters 表的 src_content 字段 │ └─ return {"parameters": all_updates, "error": None} ← 参数写入state传递给下游

输出状态：parameters: list[dict], error

节点4: param_desc_extract_node（顺序执行，等待节点3完成）

流程：

param_desc_extract_node(state) │ ├─ state.get("parameters", []) ← 从 state 读取（由节点3填充的 src_content） ├─ 过滤 src_content 非空的参数 │ ├─ 并发 LLM（asyncio.Semaphore(5)）: │ └─ PARAM_DESC_EXTRACT_PROMPT.format(param_name, src_content) │ └─ 从原文生成结构化 Markdown 表格描述 + 正则提取 direction │ ├─ MCP: update_param_descriptions(doc_id, updates) │ └─ 批量 UPDATE parameters 表的 direction、description 等字段 │ └─ return {"parameters": all_updates, "error": None} ← 参数写入state传递给下游

输出状态：parameters: list[dict], error

注意：_extract_one() 中 data_type、data_format、shape、memory_desc 仍硬编码为空字符串。其中 shape 由下游 shape_extract 填充，data_type 由 dtype_extract 填充，is_optional 由 optional_extract 填充。

节点5a: shape_extract_node（与 5b、5c 并行执行）

触发条件：param_desc_extract_node 完成后，与 dtype_extract_node、optional_extract_node 同时启动。

流程：

shape_extract_node(state) │ ├─ state.get("parameters", []) ← 从 state 读，不调 MCP 查询 ├─ 过滤 description 非空的参数 │ ├─ 逐参数并发 LLM（asyncio.Semaphore(5)）: │ └─ SHAPE_EXTRACT_PROMPT.format(param_name=..., params_text=description) │ └─ 判断前置条件（"当.../如果.../若..." → 忽略） │ └─ 无条件 → 提取 shape 值 │ └─ 返回 JSON: {"param_name": "...", "shape": "..."} │ └─ MCP: update_param_shape(doc_id, shape_updates) └─ 专用工具，SQL 只更新 shape 列

输出状态：error

节点5b: dtype_extract_node（与 5a、5c 并行执行）

触发条件：param_desc_extract_node 完成后，与 shape_extract_node、optional_extract_node 同时启动。

流程：

dtype_extract_node(state) │ ├─ state.get("parameters", []) ← 从 state 读，不调 MCP 查询 ├─ 过滤 description 非空的参数 │ ├─ 逐参数并发 LLM（asyncio.Semaphore(5)）: │ └─ DTYPE_EXTRACT_PROMPT.format(param_name=..., params_text=description) │ └─ 从 Markdown 表格 "数据类型" 行提取 │ └─ 返回 JSON: {"param_name": "...", "dtype": "..."} │ └─ 代码层 .upper() 统一为大写 │ └─ MCP: update_param_dtype(doc_id, dtype_updates) └─ 专用工具，SQL 只更新 dtype_desc 列

输出状态：error

与 5a 的关系：三个节点代码结构完全相同，仅 prompt、提取目标列、MCP 工具不同。都从 state.parameters 读取，各自写入独立列（shape / dtype_desc / is_optional），互不冲突。

节点5c: optional_extract_node（与 5a、5b 并行执行）

触发条件：param_desc_extract_node 完成后，与 shape/dtype 同时启动。

流程：

optional_extract_node(state) │ ├─ state.get("parameters", []) ← 从 state 读，不调 MCP 查询 ├─ 过滤 description 非空的参数 │ ├─ 逐参数并发 LLM（asyncio.Semaphore(5)）: │ └─ OPTIONAL_EXTRACT_PROMPT.format(param_name=..., name_has_optional=..., params_text=description) │ └─ 判断描述中是否包含可选关键词 │ └─ 参数名含 Optional → 传给 LLM 提示排除干扰 │ └─ 返回 JSON: {"param_name": "...", "is_optional": true/false} │ └─ 代码层归一化为 0/1 │ └─ MCP: update_param_optional(doc_id, optional_updates) └─ 专用工具，SQL 只更新 is_optional 列

输出状态：error

5.3 执行时序总结

gantt
    title Pipeline 执行时序
    dateFormat X
    axisFormat %s

    section 顺序
    init_doc (版本检查+保存+解析) :0, 1

    section 并行 2a/2b
    product_support (LLM 提取产品表) :1, 3
    parse_params (LLM 提取参数列表) :1, 3

    section 顺序
    src_content_extract (LLM 并发提取原文) :3, 4
    param_desc_extract (生成描述表格) :4, 5

    section 并行 5a/5b/5c
    shape_extract (逐参数提取 shape) :5, 7
    dtype_extract (逐参数提取 dtype) :5, 7
    optional_extract (逐参数判断可选) :5, 7

MCP 通信模式：每个节点每次调用 MCP 工具时，MCPClient._call_tool() 会启动一个新的 MCP Server 子进程（async with stdio_client），调用完成后进程退出。整个 Pipeline 执行一次约产生 ~16 次子进程启动。参数查询在 src_content_extract 中进行一次，下游节点通过 state 传递数据，避免重复 MCP 查询。

6. 数据库 Schema

-- 算子注册表
operators (
    id INTEGER PK,
    name TEXT NOT NULL UNIQUE,
    source_url TEXT,
    created_at TEXT
)

-- 文档版本表（每次上传新版本追加一行）
document_versions (
    id INTEGER PK,
    operator_id INTEGER FK -> operators(id),
    version INTEGER NOT NULL,
    content TEXT NOT NULL,        -- 原始 Markdown 全文
    content_hash TEXT NOT NULL,   -- SHA256
    parsed_data TEXT,             -- JSON: ParsedOperatorDocument
    product_support TEXT,         -- JSON: [{product, supported}]
    created_at TEXT,
    UNIQUE(operator_id, version)
)

-- 参数表
parameters (
    id INTEGER PK,
    doc_id INTEGER FK -> document_versions(id),
    function_name TEXT NOT NULL,
    param_name TEXT NOT NULL,
    param_type TEXT DEFAULT '',
    direction TEXT DEFAULT 'input',  -- 'input' | 'output' (param_desc_extract填充)
    src_content TEXT,                -- 原始参数描述 Markdown (src_content_extract填充) (ALTER TABLE 迁移v3添加)
    description TEXT,                -- Markdown 格式 (param_desc_extract填充)
    usage_notes TEXT,                -- 暂无提取
    dtype_desc TEXT,                 -- 数据类型 (dtype_extract填充)
    dformat_desc TEXT,               -- 暂无提取
    shape TEXT,                      -- shape (shape_extract填充)
    is_optional     INTEGER NOT NULL DEFAULT 0,  -- 0/1 (optional_extract填充) (ALTER TABLE 迁移添加)
    memory_desc TEXT,                -- 暂无提取
    created_at TEXT,
    UNIQUE(doc_id, function_name, param_name)
)

-- 索引
idx_parameters_doc_id ON parameters(doc_id)

9. 已知问题与待改进项

性能

  1. 每次工具调用启动新进程：MCPClient._call_tool() 每次创建新的 MCP Server 子进程，无连接复用，频繁启动开销大
  2. 同步 SQLite：MCP Server 全部使用同步 SQLite（aiosqlite 在依赖中声明但未使用）

代码质量

  1. _parse_json_response 重复：parse_params.py 和 product_support.py 中相同函数各一份
  2. param_desc_extract_node 字段填充不全：_extract_one() 中 usage_notes、data_format、memory_desc 仍硬编码为空字符串；parse_params.py 使用内联 prompt 而非从 prompts/__init__.py 导入
  3. 三个并行节点 Semaphore 独立：shape_extract、dtype_extract、optional_extract 各自 Semaphore(5)，并行时实际 LLM 调用峰值可达 15，可能触发 API 限流
  4. _should_continue 路由缺陷：_should_continue 仅检查 status=="error" 终止 Pipeline，status=="unchanged" 时仍扇出到 product_support + parse_params，导致不必要的 MCP 调用。应改为 status in ("error", "unchanged") 或 unchanged 时直接 END

安全

  1. .env 含真实 API Key：DeepSeek 密钥被写入 .env 文件（建议轮换并使用 Secret Manager）
  2. 未集成 bandit 扫描：CLAUDE.md 要求 bandit -r src/，但未配置

测试

  1. 测试覆盖严重不足：仅 mcp_server/parsers/ 有 34 个测试（25->34 已增加），agent 包全部无测试
  2. 无 CI/CD：没有 GitHub Actions、Dockerfile 或部署配置

文档/结构

  1. CLAUDE.md 架构图过时：描述 src/agent/、src/api/ 扁平结构，实际为 packages/*/src/ monorepo
  2. LangGraph Tools 存根：agent/tools/ 下的 @tool 定义未被节点使用（节点直接调 MCPClient）

前端

  1. 字段名不匹配 ✓ 已修复 — MCP 层 dict key 从 dtype_desc/dformat_desc 重命名为 data_type/data_format
  2. nodes/__init__.py 未同步：只导出 4 个旧节点（init_doc, parse_params, product_support, src_content_extract），未包含后续新增的 4 个节点（param_desc_extract, shape_extract, dtype_extract, optional_extract）。
  3. Markdown 渲染与双过滤器：已集成 marked.js CDN，"描述"列渲染 Markdown 格式。新增函数名下拉过滤器，与算子过滤器联动，客户端过滤。
