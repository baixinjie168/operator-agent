Operator Agent 项目总览

生成日期: 2026-05-28 | 更新日期: 2026-06-03 | 基于代码实际状态

项目定位

基于 LangGraph + MCP 的 CANN 算子文档自动化处理系统。将华为昇腾平台的算子 Markdown 文档解析为结构化数据，经 LLM 提取参数、约束、产品支持信息，最终生成 Python 测试用例。

当前完成度：约 80%，文档解析、数据库层、Pipeline 全部 18 个节点（含 src_content 提取、function_signature 提取、function_explanation 提取、shape/dtype/dformat/optional/param_attr 提取、array_length 提取、allowed_range 提取、return_code 提取、determinism 提取、dtype_combo 提取、参数关系提取子图、build_param_constraint 参数约束构建节点、assemble_result 汇编节点）可运行。人工审核、测试用例生成尚未实现。

节点1: init_doc_node（顺序执行）

数据流 & 依赖关系

上游节点START（Pipeline 入口，由 upload 路由触发）
同级节点无
下游节点_should_continue 路由 → product_support (2a) parse_params (2b) function_signature_extract (2c) function_explanation_extract (2d)
State 输入operator_name, content, content_hash（upload 路由构造的初始状态）
State 输出operator_id, doc_id, version, sections, cann_version, status, error
DB 写入operators (INSERT), document_versions (INSERT: content, content_hash, parsed_data)

触发条件：Pipeline 启动后第一个执行。

init_doc_node(state) │ ├─ MCP: check_version(operator_name, content_hash) │ └─ 查询数据库，返回 {status: "new"|"unchanged"|"updated", version, doc_id} │ ├─ 如果 status == "unchanged"（内容完全未变）: │ ├─ MCP: get_parsed(operator_name, existing_version) │ └─ 返回 {status="unchanged", doc_id, sections, cann_version, error=None} │ └─ 如果 status == "new" 或 "updated": ├─ MCP: save_doc(operator_name, content) → {operator_id, version, doc_id} ├─ MCP: parse_doc(content) → ParsedOperatorDocument ├─ MCP: save_parsed(operator_name, version, parsed_dict) └─ 返回 {status, version, doc_id, cann_version, sections, error=None}

输出状态：operator_id, doc_id, version, sections, cann_version, status, error

路由函数：_should_continue

def _should_continue(state: dict) -> list[str]:
    if state.get("status") in ("error",):
        return END                    # 终止 Pipeline
    return ["product_support", "parse_params", "function_signature_extract", "function_explanation_extract"]  # 扇出到四个并行节点
⚠ status=="unchanged" 时仍扇出到并行节点，导致不必要的 MCP 重复调用。应在此处加入 "unchanged" 判断直接终止 Pipeline。

MCP 通信模式：每个节点每次调用 MCP 工具时，MCPClient._call_tool() 会启动一个新的 MCP Server 子进程（async with stdio_client），调用完成后进程退出。整个 Pipeline 执行一次约产生 ~45 次子进程启动（param_relation_extract 子图额外增加 3~5 次，allowed_range_extract 额外增加 3~6 次，return_code_extract 额外增加 3~6 次，determinism_extract 额外增加 2~3 次，dtype_combo_extract 额外增加 2~3 次，assemble_result 额外增加 8 次）。参数查询在 src_content_extract 中进行一次，下游节点通过 state 传递数据，避免重复 MCP 查询。
