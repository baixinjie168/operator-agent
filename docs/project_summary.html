生成日期: 2026-05-28 | 更新日期: 2026-06-08 (数据质量修复: dimensions 符号维度 rank 解析 + dtype key 不匹配 + param_constraint.type const 限定符剥离 + function_signature 后处理规范化) | 基于代码实际状态

项目定位

基于 LangGraph + MCP 的 CANN 算子文档自动化处理系统。将华为昇腾平台的算子 Markdown 文档解析为结构化数据，经 LLM 提取参数、约束、产品支持信息，最终生成 Python 测试用例。

当前完成度：约 93%，文档解析、数据库层、Pipeline 全部 18 个节点（含 llm_description_extract 子图（预过滤 + 属性校验）、function_signature 提取（含参数类型后处理规范化）、function_explanation 提取、shape/dtype/dformat/optional/array_length 提取、allowed_range 提取、return_code 提取、determinism 提取、dtype_combo 提取、参数关系提取子图（Agent 迭代循环：分块提取 + 双重覆盖率 + 针对性提取 + 自反思）、build_param_relations 参数关系表达式构建节点、build_param_constraint 参数约束构建节点（含类型规范化 + dimensions rank 解析修复）、assemble_result 汇编节点 + json_constraints 写入）可运行。数据质量问题（dtype key 不匹配、const 限定符污染、符号维度解析错误）已修复。人工审核、测试用例生成尚未实现。

init_doc_node(state) │ ├─ MCP: check_version(operator_name, content_hash) │ └─ 查询数据库，返回 {status: "new"|"unchanged"|"updated", version, doc_id} │ ├─ 如果 status == "unchanged"（内容完全未变）: │ ├─ MCP: get_parsed(operator_name, existing_version) │ └─ 返回 {status="unchanged", doc_id, sections, cann_version, error=None} │ └─ 如果 status == "new" 或 "updated": ├─ MCP: save_doc(operator_name, content) → {operator_id, version, doc_id} ├─ MCP: parse_doc(content) → ParsedOperatorDocument ├─ MCP: save_parsed(operator_name, version, parsed_dict) └─ 返回 {status, version, doc_id, cann_version, sections, error=None}

路由函数：_should_continue

def _should_continue(state: dict) -> list[str]:
    if state.get("status") in ("error",):
        return END                    # 终止 Pipeline
    return ["product_support", "parse_params", "function_signature_extract", "function_explanation_extract"]  # 扇出到四个并行节点
⚠ status=="unchanged" 时仍扇出到并行节点，导致不必要的 MCP 重复调用。应在此处加入 "unchanged" 判断直接终止 Pipeline。

子图内部节点详解

fetch_sections

从 parameters 中按 function_name 分为 ws/exe 两组，仅对有参数的组调用 MCP 获取 section 内容，避免不必要的 MCP 调用。

extract_ws / extract_exe

两个节点并行执行，共享 _extract_one() 核心逻辑（定义在 extract_ws.py，被 extract_exe.py 导入复用）。核心优化：

  • 预过滤上下文：extract_param_context(sections_text, param_name) 将完整 section 文本（3000-12500 字符）缩减为仅与该参数相关的片段（500-2000 字符），减少 LLM 注意力分散
  • 并发控制：asyncio.Semaphore(5) 限制同时进行的 LLM 调用数
  • 一次 LLM 调用提取四个字段：llm_description + src_content + direction + is_support_discontinuous

validate_results

对提取结果执行属性检查清单校验（attribute_checklist_verify），检查 llm_description 是否覆盖了上下文中实际存在的属性（direction/dtype/shape/optional/discontinuous）。同时生成覆盖率报告（coverage_report），统计提取成功数、缺失数、描述过短数等指标。

save_descriptions

清理内部字段（_context, _validation）后通过 MCP 批量 UPDATE 数据库，然后将 enriched parameters 回写到 parameters 供主图下游节点消费。

  1. Phase 1 - 确定性预处理：基于正则的零 LLM 成本解析
    • dimensions：支持三种输出格式
      • rank exact：[N, N] - 精确 N 维（如 "2D" → [2, 2]）
      • rank range：[min_rank, max_rank] - 支持 min 到 max 维（如 "0-8" → [0, 8]，"1-8" → [1, 8]）
      • 逐维范围：[[min,max], ...] - 每维的 size 范围（如 "[2, 3, 4]" → [[2,2],[3,3],[4,4]]）
    • allowed_range_value：常见模式直接提取
      • 范围："0-100" / "[1, 8]" → [[0, 100]] / [[1, 8]]
      • 枚举："枚举值: 1, 2, 3" → [[1,1], [2,2], [3,3]]
      • bool 类型短路：[True, False]
  2. Phase 2 - LLM 解析 + 校验 + 重试
    • 确定性预处理未匹配的 shape 进入 LLM 批量解析（SHAPE_TO_DIMENSIONS_PROMPT）
    • 确定性预处理未匹配的 allowed_range_value 进入 LLM 逐参数提取（ALLOWED_RANGE_VALUE_BUILD_PROMPT）
    • 结构校验：dimensions 校验 rank format（0 ≤ min_rank ≤ max_rank ≤ 10）和逐维范围（min ≤ max）；allowed_range_value 校验 min ≤ max + 类型匹配 + 合理性（不超过 10^9）
    • 对齐校验（dimensions 专用）：LLM 返回数组长度必须与输入 shape 数量一致
    • 溯源校验（allowed_range_value 专用，软校验）：检查提取值是否在源文本中出现
    • 失败补救：校验失败时最多重试 2 次，仍失败则降级为单条目 LLM 调用（dimensions）或返回空数组（allowed_range_value）

MCP 通信模式：每个节点每次调用 MCP 工具时，MCPClient._call_tool() 会启动一个新的 MCP Server 子进程（async with stdio_client），调用完成后进程退出。整个 Pipeline 执行一次约产生 ~50 次子进程启动（llm_description_extract 子图额外增加 5~7 次 [fetch + N 个参数的 update]，param_relation_extract 子图额外增加 3~5 次，allowed_range_extract 额外增加 3~6 次，return_code_extract 额外增加 3~6 次，determinism_extract 额外增加 2~3 次，dtype_combo_extract 额外增加 2~3 次，build_param_relations 额外增加 3~5 次，build_param_constraint 额外增加 5~8 次，assemble_result 额外增加 10 次）。参数列表在 parse_params 中创建，llm_description_extract 充实后通过 state 传递给下游，避免重复 MCP 查询。
