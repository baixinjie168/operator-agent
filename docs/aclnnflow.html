解析过程中发现的问题

3个问题
问题 1: target/weightOptional/posWeightOptional 的 dtype 值为 "与self一致" 而非实际类型列表

现象: 在最终JSON中，target/weightOptional/posWeightOptional的dtype.value为 ["与self一致"]，而非 ["BFLOAT16","FLOAT","FLOAT16"]

原因: table_parser 的相对引用检测机制在Phase 3跳过了"与self一致"的值，后续DtypeExtract阶段LLM也未能将其解析为具体类型列表，而是将原始引用文本直接作为dtype值

影响: dtype字段应该是具体的类型枚举列表(如["FLOAT16","FLOAT","BFLOAT16"])，而非引用文本。正确的处理应通过ParamRelation(type_equality: target.dtype == self.dtype)来表达类型一致性，同时在dtype.value中填入与self相同的实际类型列表

涉及位置: JSON L769-771 (target dtype), L930-932 (weightOptional dtype), L270-272 (posWeightOptional dtype)

问题 2: out 参数的 dtype 和 format 均为空列表

现象: out的dtype.value = [] (空) 和 format.value = [] (空)

原因: Markdown参数表中out行的"数据类型"和"数据格式"列均为"-"(dash)，且无"与self一致"的显式引用。虽然"使用说明"中提到"数据格式需要与self一致"，但table_parser只提取6列固定字段，"使用说明"列的内容由LLM描述节点处理

影响: out的dtype应为与self一致的具体类型列表，format应为["ND"]。当前约束表达式中 out.format == self.format 表达了格式一致性，但out自身的format.value为空，无法直接知道out支持什么格式

涉及位置: JSON L1092-1094 (out dtype), L1071-1073 (out format)

问题 3: dimensions.value 为空列表但 src_text 有值

现象: self/weightOptional/posWeightOptional/out 的 dimensions.value = []，但 dimensions.src_text = "ND" 或 "与self一致"

原因: dimensions.value = [] 表示"无法确定维度约束"。当Markdown表格的"维度(shape)"列为 "-" 时，LLM无法推断出具体的维度范围。src_text保留了原始引用("ND"来自format列被误用，"与self一致"来自shape列)

影响: dimensions.value=[] 对于self这类主参数是合理的(支持任意维度)，但src_text中"ND"是format而非shape的描述，存在语义混淆

涉及位置: JSON L594-596 (self dimensions.src_text="ND"), L226-228 (posWeightOptional dimensions.src_text="ND")
